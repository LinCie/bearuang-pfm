# Decision Log — bearuang Backend API PRD

## Decisions

### D-001: Backend-only scope
**Date:** 2026-05-27
**Decision:** PRD covers only the backend API layer (Hono on Cloudflare Workers). Frontend/PWA is a separate concern.
**Rationale:** User explicitly requested backend-only PRD. Allows focused implementation without coupling to UI decisions.

### D-002: MVP feature set derived from brainstorming Tier 1
**Date:** 2026-05-27
**Decision:** All 25 Tier 1 must-have features from the brainstorming session are represented in functional requirements. Select Tier 2 items (upcoming payments list, soft delete, monthly cash flow) included where they have minimal backend cost and high API completeness value.
**Rationale:** Tier 1 represents the minimum viable product as agreed during brainstorming. A few Tier 2 items are essentially free once the data model exists.

### D-003: Hono framework retained
**Date:** 2026-05-27
**Decision:** Use Hono as the HTTP framework, already present in the project.
**Rationale:** Already in package.json, lightweight, Cloudflare Workers native, good TypeScript support.

### D-004: Password auth over passkey for MVP
**Date:** 2026-05-27
**Decision:** Simple password authentication with env-variable bootstrap. Passkey deferred.
**Rationale:** Confirmed in brainstorming as simpler for v1. Passkey adds WebAuthn complexity without clear single-user benefit.

### D-005: Flat categories only
**Date:** 2026-05-27
**Decision:** No hierarchy, no tags. Simple flat list of categories.
**Rationale:** Brainstorming explicitly deferred hierarchical categories and tags as overkill for MVP.

### D-006: Multi-user data model from day one
**Date:** 2026-05-27
**Decision:** Schema includes user_id foreign keys on all mutable entities even in solo mode.
**Rationale:** Brainstorming confirmed collaboration-ready data model to avoid painful migration later.

### D-007: Client-generated UUIDs for offline support
**Date:** 2026-05-27
**Decision:** Transactions use client-generated UUIDs as primary keys with server-side uniqueness enforcement.
**Rationale:** Enables offline-first PWA to create transactions locally and sync without server round-trip for ID generation.

### D-008: No CSV/OFX import in MVP
**Date:** 2026-05-27
**Decision:** Import functionality deferred to post-MVP. Export is MVP.
**Rationale:** Import requires column mapping UI, duplicate detection, preview — significant complexity. Export is simpler and satisfies data ownership promise.

### D-009: Activity log for collaboration
**Date:** 2026-05-27
**Decision:** Append-only activity log recording who did what. Not a full audit trail with before/after values.
**Rationale:** Supports partner coordination without the complexity of full event sourcing. Full audit trail is post-MVP.

### D-010: Single currency per instance for MVP
**Date:** 2026-05-27
**Decision:** One base currency configured at setup. No multi-currency, no exchange rates.
**Rationale:** Multi-currency with manual rates was Tier 2. Keeping single currency simplifies all aggregation queries.
